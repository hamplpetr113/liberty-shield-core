# Runtime Boundary Contract — Controlled Chaos Engine

**Version:** 1.0  
**Status:** Authoritative — Sprint 7 network integration must satisfy all conditions herein before any socket-touching code is written.  
**Scope:** Defines the strict interface between the Controlled Chaos Engine and every layer beneath it, from StreamMux down to UDP Transport.

---

## 1. Purpose

The Controlled Chaos Engine produces *scheduling intent*: it decides which packets should be sent, on which paths, at which times, and with what priority. It has no authority over — and no visibility into — the networking stack that carries out those intents.

This boundary exists for three reasons:

**Security isolation.** The Controlled Chaos Engine operates on metadata and encrypted payload references. It must never be able to emit cleartext traffic, bypass the kill switch, or reach Android networking APIs directly. If the engine is compromised or misconfigured, the boundary prevents that failure from becoming a network-level vulnerability.

**Auditability.** Every packet that leaves the device must pass through a single validation checkpoint. Centralising validation in one place (the `RuntimeBoundaryValidator`) makes it possible to audit, log, and test network egress behaviour exhaustively without reasoning about the internal state of six separate subsystems.

**Deterministic testability.** The engine must be fully testable without a network stack. All outputs are pure data structures. Tests can inject any `ControlledChaosOutput` and assert on `RuntimeValidationResult` without a socket, a VPN service, or an Android device in the loop.

---

## 2. Allowed Outputs from Controlled Chaos

The Controlled Chaos Engine may produce exactly one output type: a `ControlledChaosOutput`. This value is a scheduling intent, not a packet. It carries only the metadata required for downstream layers to route, encode, and transmit — it contains no decrypted payload bytes and no addressing information sufficient to bypass the VPN tunnel.

### 2.1 Fields of `ControlledChaosOutput`

| Field | Type | Description |
|---|---|---|
| `path_id` | `u64` | Identifies the logical path (mapped to a tunnel endpoint by `StreamMux`). Controlled Chaos never resolves this to an IP address or socket descriptor. |
| `flow_id` | `u64` | Session-scoped flow identifier. Opaque to all layers below `StreamMux`. |
| `fragment_id` | `u64` | Monotonically increasing per-flow sequence number. Even = real packet, odd = cover/shadow packet. |
| `scheduled_send_time` | `u64` (μs, monotonic) | The earliest time at which this intent may be dispatched. Layers below must not send before this time. They may send later only within `latency_deadline`. |
| `packet_class` | `PacketClass` | One of `Real`, `Shadow`, `Cover`. Determines drop policy on validation failure (see §6). |
| `shadow_flag` | `bool` | `true` when this intent was generated by `ShadowSyncEngine` rather than by a real-packet arrival. Redundant with `packet_class == Shadow` but included explicitly to prevent class-mismatch bugs. |
| `latency_deadline` | `u64` (μs, monotonic) | Absolute deadline. If the intent has not been dispatched by this time, it must be dropped (silently for shadow, with error for real). |
| `payload_ref` | `PayloadRef` | A validated handle to an encrypted, already-framed payload buffer. See §2.2. |

### 2.2 `PayloadRef` invariants

- `PayloadRef` is an index into an owned, caller-managed buffer pool. It is not a raw pointer.
- The buffer it references contains only the encrypted, encapsulated form of the payload — no plaintext is accessible via this handle.
- The buffer pool is owned by the layer above the Controlled Chaos Engine. `ControlledChaosOutput` borrows a reference; it cannot extend the lifetime of the buffer or cause it to be re-used after release.
- For shadow/cover packets, `PayloadRef` points to PRNG-generated bytes produced by `ShadowSyncEngine::synthetic_payload`. These bytes have no relationship to any real user payload.

## 3. Forbidden Behavior

The following are hard prohibitions. No future sprint, performance optimisation, or emergency patch may relax these constraints without a full security review.

**The Controlled Chaos Engine must never:**

- Open, read, write, or poll any socket (TCP, UDP, Unix domain, or otherwise).
- Send a UDP packet or any network datagram directly, at any layer.
- Call any Android API — including but not limited to `VpnService`, `ConnectivityManager`, `NetworkCallback`, `InetAddress`, or `Binder`.
- Bypass `FirewallController`. Every packet intent must pass through `FirewallController` policy evaluation before reaching `StreamMux`.
- Bypass `TunnelState`. The engine must not cache, infer, or act on tunnel state directly; it observes tunnel state only via `PathEvent` inputs delivered to it from above.
- Generate cleartext network output. The engine operates exclusively on encrypted payload references (§2.2).
- Override or ignore `KillSwitch` state. The engine does not read `KillSwitch` directly; `RuntimeBoundaryValidator` enforces this unconditionally before any intent is processed.
- Inspect payload contents. The engine must treat all `PayloadRef` values as opaque handles. It must not decode, hash, or pattern-match on payload bytes.

**Additionally, the following are forbidden at the type level** — violations must be caught at compile time, not runtime:

- `ControlledChaosOutput` must not contain any type that gives access to a socket, file descriptor, or OS handle.
- `ControlledChaosOutput` must not contain a raw `*const u8` or `*mut u8` payload pointer — only `PayloadRef`.
- The `liberty-controlled-chaos` crate must not depend on any networking crate (`tokio`, `mio`, `socket2`, `nix` networking modules, Android NDK networking bindings, etc.).

## 4. Security Hierarchy

The following priority order is absolute. Higher layers may block, drop, or modify the decisions of lower layers. Lower layers must not circumvent or assume the absence of controls above them.

```
KillSwitch                 ← highest authority; blocks all egress unconditionally
  │
FirewallController         ← per-flow, per-destination policy enforcement
  │
TunnelState                ← VPN tunnel liveness; all traffic requires TunnelUp
  │
SessionAnchoring           ← session key validity; no transmission on expired session
  │
ControlledChaos            ← scheduling intent generation (this engine)
  │
StreamMux                  ← path multiplexing; maps path_id to tunnel sub-stream
  │
CellEncoder                ← framing, padding, and cell-level encryption
  │
NoiseLink                  ← Noise protocol handshake and record-layer encryption
  │
UDPTransport               ← lowest layer; writes datagrams to the OS socket
```

**Rules derived from this hierarchy:**

1. A `KillSwitch` activation must cause `RuntimeBoundaryValidator` to reject all intents with no exception — including intents already queued in `FragmentQueue` or pending in `Scheduler`.
2. `FirewallController` policy is evaluated *after* Controlled Chaos scheduling and *before* `StreamMux`. A flow that Controlled Chaos marks as `Real` may still be blocked by firewall policy; Controlled Chaos does not receive feedback about this and must not retry autonomously.
3. `TunnelState` transitions are communicated to Controlled Chaos as `PathEvent::PathDown` / `PathEvent::PathRestored`. Controlled Chaos reacts by emitting `PlanInvalidationEvent`; it does not act on `TunnelState` directly.
4. No layer below `ControlledChaos` in the hierarchy may promote a `Shadow` packet to `Real`, change `path_id`, extend `latency_deadline`, or alter `payload_ref`.

## 5. Runtime Validation

Before any `ControlledChaosOutput` may enter `StreamMux`, the `RuntimeBoundaryValidator` must evaluate all of the following conditions in the listed order. The first failing check determines the failure mode (§6).

| # | Check | Failure mode |
|---|---|---|
| V1 | KillSwitch is inactive | Fail closed — drop all, log |
| V2 | TunnelState is `TunnelUp` | Fail closed — drop all, log |
| V3 | `path_id` exists in the current active path set | Emit `PlanInvalidationEvent` |
| V4 | `latency_deadline` has not elapsed (`now_us ≤ latency_deadline`) | Drop (class-dependent; see §6) |
| V5 | `payload_ref` is valid (buffer not released, length in [64, 1500]) | Drop real, drop shadow; log corruption |
| V6 | Shadow packet budget not exceeded for this flow in this epoch | Drop shadow silently |

**Ordering rationale:**

- V1 and V2 are checked first because they are global invariants. If either fails, no further evaluation is warranted.
- V3 precedes V4 because an invalid path cannot produce a meaningful deadline calculation.
- V5 precedes V6 because a corrupted `PayloadRef` must not be accounted against the shadow budget.

**Shadow packet budget definition:**

The shadow budget for flow `f` in epoch `e` is the maximum number of cover/shadow bytes permitted to be transmitted in one epoch, as computed by `build_fragment_plan` (specifically `total_cover_bandwidth_kbps` × epoch duration). Exceeding this budget must not cause real packets to be dropped; only shadow packets are subject to budget enforcement.

## 6. Failure Behavior

The table below defines the exact response to each failure category. "Fail closed" means the packet does not proceed and no retry is attempted by the boundary layer.

| Condition | Real packet | Shadow packet |
|---|---|---|
| KillSwitch active | Drop; log; return `Err(KillSwitchActive)` to forwarder | Drop silently |
| TunnelState not `TunnelUp` | Drop; log; return `Err(TunnelDown)` to forwarder | Drop silently |
| Invalid `path_id` | Drop; emit `PlanInvalidationEvent { reason: PathDown { path_id } }`; forwarder triggers replan | Drop silently |
| Deadline expired (`latency_deadline` elapsed) | Drop; log with late-by duration; return `Err(DeadlineMissed)` to forwarder | Drop silently |
| Invalid `payload_ref` | Drop; log as internal error (possible memory safety issue); return `Err(PayloadRefInvalid)` | Drop silently; log as internal error |
| Shadow budget exceeded | Not applicable — real packets are not subject to shadow budget | Drop silently; increment dropped-shadow counter in `TransmitterStats` |

**General principle:** real packets are never silently dropped. Every dropped real packet produces a logged error and an error value returned to the caller. Shadow packets are always dropped silently — their loss must not trigger retransmission, replan, or user-visible error.

**Replan trigger:** When a real packet is dropped due to `InvalidPath`, the `PlanInvalidationEvent` carries the affected `path_id` and the current `PathStats`. The layer above Controlled Chaos (the VPN forwarder) is responsible for invoking `build_fragment_plan` with updated candidates. Controlled Chaos does not self-replan.

## 7. Required Rust Interface

The following types define the boundary contract in Rust. These are *type proposals only* — no implementation is written in this document. Sprint 7 must implement these types exactly as specified before any networking code may be written.

```
/// The sole output type of the Controlled Chaos Engine.
/// Contains scheduling intent only — no socket handles, no raw pointers,
/// no plaintext payload bytes.
struct ControlledChaosOutput {
    path_id:             u64,
    flow_id:             u64,
    fragment_id:         u64,
    scheduled_send_time: u64,   // μs, monotonic
    packet_class:        PacketClass,
    shadow_flag:         bool,
    latency_deadline:    u64,   // μs, monotonic
    payload_ref:         PayloadRef,
}

enum PacketClass {
    Real,
    Shadow,
    Cover,
}

/// Opaque handle into the buffer pool.  Must not expose raw pointer access.
struct PayloadRef {
    pool_index: u32,
    length:     u16,
    // No pub fields — accessed only through the pool's checked API.
}

/// The value passed to StreamMux after successful validation.
/// Structurally identical to ControlledChaosOutput but carries the
/// type-system proof that RuntimeBoundaryValidator has accepted it.
struct RuntimePacketIntent {
    inner: ControlledChaosOutput,
    // Private field prevents construction outside RuntimeBoundaryValidator.
}

/// The result of boundary validation.
enum RuntimeValidationResult {
    Accept(RuntimePacketIntent),
    Reject(RejectionReason),
}

enum RejectionReason {
    KillSwitchActive,
    TunnelDown,
    PathNotFound     { path_id: u64 },
    DeadlineMissed   { path_id: u64, late_by_us: u64 },
    PayloadRefInvalid,
    ShadowBudgetExceeded { flow_id: u64 },
}

/// The single validation checkpoint between Controlled Chaos and StreamMux.
/// Constructed with references to KillSwitch, TunnelState, and the active
/// path set.  Never holds mutable state itself.
struct RuntimeBoundaryValidator {
    // Fields are not specified here — Sprint 7 determines the concrete
    // lifetime and ownership model for KillSwitch / TunnelState references.
}

impl RuntimeBoundaryValidator {
    fn validate(
        &self,
        output:  ControlledChaosOutput,
        now_us:  u64,
        budget:  &ShadowBudgetTracker,
    ) -> RuntimeValidationResult;
}
```

**Design constraints for Sprint 7 implementors:**

- `RuntimePacketIntent` must be constructable only inside `RuntimeBoundaryValidator::validate`. The private field enforces this at the type level — `StreamMux` cannot fabricate a `RuntimePacketIntent` without going through validation.
- `RuntimeBoundaryValidator` must be `Send + Sync`. It holds no mutable state; all state it inspects (kill switch, tunnel state, path set) is accessed through shared references or atomics.
- `ShadowBudgetTracker` is a separate type owned by the calling layer. `RuntimeBoundaryValidator` reads it; it does not own it.
- No `unsafe` block is permitted in `RuntimeBoundaryValidator` or any type in this interface.

## 8. Tests Required Before Sprint 7

The following tests must exist and pass before any Sprint 7 networking code may be merged. Tests are to be written against `RuntimeBoundaryValidator` only — no test may open a socket or call any Android API.

| ID | Test name | What it asserts |
|---|---|---|
| B1 | `kill_switch_blocks_all_intents` | With kill switch active, `validate()` returns `Reject(KillSwitchActive)` for every `PacketClass` — `Real`, `Shadow`, and `Cover`. No intent proceeds to `StreamMux`. |
| B2 | `tunnel_down_blocks_all_intents` | With `TunnelState` not `TunnelUp` (covering `TunnelConnecting`, `TunnelFailed`, `TunnelTeardown`), `validate()` returns `Reject(TunnelDown)` for all packet classes. |
| B3 | `invalid_path_rejected` | A `ControlledChaosOutput` referencing a `path_id` not in the active path set returns `Reject(PathNotFound { path_id })`. A `PlanInvalidationEvent` is emitted. |
| B4 | `shadow_packet_over_budget_dropped` | When `ShadowBudgetTracker` reports budget exhausted for a flow, a `Shadow`-class intent returns `Reject(ShadowBudgetExceeded)`. A simultaneous `Real`-class intent for the same flow is not affected. |
| B5 | `real_packet_never_silently_dropped` | Across all rejection reasons (`KillSwitchActive`, `TunnelDown`, `PathNotFound`, `DeadlineMissed`, `PayloadRefInvalid`), a `Real`-class intent always returns an explicit `Reject` variant — never `Accept` and never `panic`. The caller always receives an actionable result. |
| B6 | `no_direct_udp_send_possible_from_controlled_chaos` | The `liberty-controlled-chaos` crate has no dependency path to any type that can write to a socket. This is a **cargo dependency audit test**: parse the output of `cargo tree -p liberty-controlled-chaos` and assert that none of `{tokio, mio, socket2, nix, std::net, std::os::unix::net, android_ndk}` appear in the dependency graph. |

**Notes on B6:** This test is a static analysis test, not a runtime test. It should be implemented as a build-time check or a test that shells out to `cargo tree`. The intent is to make the prohibition in §3 machine-verifiable, not just contractual.
