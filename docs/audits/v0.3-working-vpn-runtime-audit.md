## Detailed Fix Notes

### F1 — TcpSession.startServerReader finally block

**Before:**
```kotlin
} catch (e: CancellationException) {
    throw e
} catch (_: Exception) { }
if (!firstByteReceived && connectDoneMs > 0L) VpnStats.tcpSessionsNoFirstByte.incrementAndGet()
withContext(NonCancellable) { sessionMutex.withLock { teardown() } }
```

**After:**
```kotlin
} catch (e: CancellationException) {
    throw e
} catch (_: Exception) {
} finally {
    if (!firstByteReceived && connectDoneMs > 0L) VpnStats.tcpSessionsNoFirstByte.incrementAndGet()
    withContext(NonCancellable) { sessionMutex.withLock { teardown() } }
}
```

The `finally` block runs on every coroutine exit path. `NonCancellable` inside `finally` allows `teardown()` to complete even when the coroutine is being cancelled. `teardown()` is idempotent (CAS guard), so calling it twice (e.g., backpressure path + finally) is safe.

## PacketPool Safety Verification

All acquire/release paths confirmed correct:

| Path | Outcome |
|------|---------|
| Normal TUN write | `acquire()` → `bufs[i] = buf` → Phase 3 `trySend` success → `bufs[i] = null` → `writeAndRelease` finally → `release()` ✓ |
| TUN writer exception during write | `writeAndRelease` finally always calls `release(buf)` ✓ |
| Failed `trySend` to DATA queue | `bufs[i]` not nulled → outer `finally { for (buf in bufs) if (buf != null) PacketPool.release(buf) }` ✓ |
| Exception in Phase 2 (packet build) | Same outer finally releases all non-null slots ✓ |
| VPN shutdown: DATA queue stranded | Writer's `finally` drains DATA queue and releases pool buffers ✓ |
| VPN shutdown: CONTROL queue stranded | Control queue not drained — all control packets use `fromPool=false`, no pool leak ✓ |
| `CancellationException` in serverJob | `finally` block in `startServerReader` (after F1 fix) handles remaining pool buffers via `teardown()` path ✓ |

No pool leaks or double-releases found.
