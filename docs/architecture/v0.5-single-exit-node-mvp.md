# Liberty Shield v0.5 — Single Exit Node MVP

## 1. Current Stable Baseline

The Android app (v0.3.x) is a **local VPN relay**. It intercepts all IP traffic
from the phone via `VpnService`/TUN, processes it through `PacketForwarder`, and
forwards each TCP connection and DNS/UDP request **directly to the internet** using
normal Android sockets that bypass the TUN (protected via `VpnService.protect()`).

What this means:
- The phone's traffic is captured and re-relayed locally.
- Every outbound TCP/UDP socket still originates from the phone's own IP.
- The phone's public IP **does not change**.
- No encryption is added between phone and destination.
- There is no exit node.

This is a valid and stable foundation for the next phase, but it is not a VPN in
the privacy/anonymity sense. It is a transparent local relay with full telemetry
visibility.

---

## 2. Why a VPS Exit Node Is Required

`VpnService` gives the app control over all IP packets leaving the phone, but those
packets still ultimately exit through the phone's own mobile/Wi-Fi uplink unless
they are tunnelled to a remote server first.

To change the phone's observable public IP:

```
Phone uplink (own IP visible) → tunnel → VPS → internet (VPS IP visible)
```

The VPS performs NAT: it receives encapsulated traffic from the phone, decapsulates
it, and forwards it with its own IP. Return traffic is routed back through the
tunnel to the phone.

Without a VPS:
- public IP stays the phone's IP
- no encryption between phone and internet
- no exit node protection

A single exit node is the minimum viable unit of real VPN protection and is the
required first step before multi-hop or route rotation.

## 3. Target v0.5 Architecture

```
┌─────────────────────────────────────────────────────────┐
│  Android Device                                         │
│                                                         │
│  App traffic (TCP/UDP/DNS)                              │
│       ↓                                                 │
│  VpnService TUN fd                                      │
│       ↓                                                 │
│  PacketReader → PacketForwarder                         │
│       ↓                                                 │
│  TunnelClient (future)                                  │
│   • framing + encryption (Noise handshake TBD)          │
│   • UDP transport to exit node                          │
└───────────────┬─────────────────────────────────────────┘
                │  encrypted UDP tunnel
                ↓
┌─────────────────────────────────────────────────────────┐
│  VPS — Liberty Exit Node                                │
│                                                         │
│  UDP listener (port configurable, default 51820)        │
│       ↓                                                 │
│  client session registry                                │
│       ↓                                                 │
│  packet ingress / decapsulation                         │
│       ↓                                                 │
│  NAT / IP forwarding (kernel, nftables)                 │
│       ↓                                                 │
│  public internet (exit IP = VPS IP)                     │
│                                                         │
│  health endpoint (HTTP, internal only)                  │
│  metrics / structured logging                           │
└─────────────────────────────────────────────────────────┘
```

## 4. Components

### Android Side

| Component | Status | Role |
|---|---|---|
| `ShieldVpnService` | stable | VPN lifecycle, TUN establish/revoke |
| `PacketReader` | stable | reads IP packets from TUN fd |
| `PacketForwarder` | stable | dispatches TCP sessions + UDP/DNS |
| `TunnelClient` | **future** | encrypts packets, sends to exit node via UDP |
| `TunnelMode` | **future** | `LOCAL_ONLY` / `SINGLE_EXIT_NODE` enum |
| Dashboard tunnel fields | **future** | tunnel state, exit IP, bytes in/out |

`TunnelClient` sits between `PacketForwarder`'s UDP output and the real network.
In `SINGLE_EXIT_NODE` mode, all packets are encapsulated and sent to the exit node
instead of being forwarded directly. In `LOCAL_ONLY` mode (current default), the
existing forwarder path is used unchanged.

### Server Side

| Component | Status | Role |
|---|---|---|
| UDP listener | **to build** | receives encapsulated packets from clients |
| client session registry | **to build** | maps client identity → session state |
| packet ingress | **to build** | decapsulates, validates, replays-checks |
| NAT / IP forwarding | **to configure** | kernel `ip_forward` + nftables masquerade |
| health endpoint | **to build** | `GET /health` on localhost |
| metrics / logging | **to build** | structured logs, packet counts, errors |
| Noise handshake | **placeholder** | session key establishment (not in MVP) |

## 5. MVP Scope

v0.5 must prove the following end-to-end:

1. Phone connects to one configured VPS exit node.
2. Traffic is routed through the VPS (phone IP hidden from internet).
3. A public IP check from the phone shows the VPS IP, not the phone IP.
4. Dashboard shows tunnel connected / disconnected state.
5. Fallback to current `LOCAL_ONLY` mode works if no VPS is configured.
6. VPN stability from v0.3.x is not regressed.

The MVP tunnel transport is intentionally minimal:
- simple framing (length prefix + payload)
- symmetric pre-shared key encryption (AES-GCM or ChaCha20-Poly1305, key provisioned
  out-of-band; NOT hardcoded in APK)
- replay protection via nonce/counter
- Noise XX handshake is a placeholder for v0.6+

---

## 6. Non-Goals for v0.5

The following are explicitly out of scope and must not be partially implemented:

- Multi-hop routing
- Route rotation or path diversity
- Deception / cover traffic
- Onion routing
- Production anonymity claims of any kind
- Post-quantum cryptography
- Advanced node selection or directory authority
- Persistent QUIC relay (separate track)
- Full Noise XX handshake (v0.6 target)

---

## 7. Security Boundaries

These rules apply to all v0.5 code and must not be violated:

| Boundary | Rule |
|---|---|
| APK is untrusted | Treat APK as fully reverse-engineerable. No long-term secrets. |
| No static private keys in APK | Exit node keys live on VPS only. |
| No hardcoded pre-shared secrets | PSK provisioned via separate device enrollment (TBD). |
| No plaintext credentials in source | No passwords, tokens, or keys in any source file. |
| Server config stays on VPS | `LIBERTY_EXIT_BIND`, keys, allowed peers — VPS env only. |
| Logs must not store payloads | Log packet counts and sizes; never log payload content. |
| VPN status must be visible | User must always be able to see tunnel on/off in dashboard. |
| No stealth or hidden behavior | All VPN behavior must be transparent to the user. |
| User consent required | VPN starts only on explicit user action; Android `VpnService.prepare()` enforced. |

Device provisioning (how a phone gets a PSK authorised for a specific exit node) is
a separate track and must be designed before v0.5 ships to any real user.

## 8. Phase 4 — Future Android Integration Plan

The following Android changes are planned but NOT implemented in this mission.

### TunnelMode enum

```kotlin
enum class TunnelMode {
    LOCAL_ONLY,          // current default; PacketForwarder connects directly
    SINGLE_EXIT_NODE,    // all traffic routed through TunnelClient → VPS
}
```

`LOCAL_ONLY` must always work even if no VPS is configured. The mode is controlled
by a runtime config value, not a compile-time flag.

### TunnelClient interface (sketch)

```kotlin
interface TunnelClient {
    suspend fun connect(host: String, port: Int)
    suspend fun sendPacket(data: ByteArray, len: Int)
    suspend fun receivePacket(): ByteArray
    fun disconnect()
    val isConnected: Boolean
}
```

`PacketForwarder` delegates to `TunnelClient` in `SINGLE_EXIT_NODE` mode instead
of dispatching to `TcpSession` / `forwardUdp`.

### Future Dashboard Fields

| Field | Description |
|---|---|
| `tunnelMode` | LOCAL_ONLY / SINGLE_EXIT_NODE |
| `exitNodeHost` | configured exit node hostname or IP |
| `tunnelConnected` | boolean; live connection state |
| `tunnelBytesOut` | cumulative bytes sent to exit node |
| `tunnelBytesIn` | cumulative bytes received from exit node |
| `publicIpCheckStatus` | UNKNOWN / CHECKING / CONFIRMED / MISMATCH |
| `exitPublicIp` | last confirmed public IP via exit node |

### Default Mode

`TunnelMode.LOCAL_ONLY` is the default at compile time and runtime. No VPS config
= no tunnel attempt. The existing relay path is untouched.

### No Hardcoded Server IP

Server address is loaded from a provisioned config, not from `BuildConfig` or
source. A debug/test override via `BuildConfig.DEBUG_EXIT_NODE_HOST` is acceptable
for dev builds only and must be absent from release builds.

## 9. Phase 5 — Public IP Check (Debug Only)

A manual, on-demand public IP check will be added to the dashboard:

- Trigger: "Check Public IP" button (visible in debug builds or when tunnel mode is active).
- Behaviour: single HTTP GET to a configurable endpoint (default: `https://api.ipify.org`
  or equivalent; endpoint is not hardcoded as a permanent dependency).
- Result display:
  - VPN off: phone's real IP
  - VPN on, `LOCAL_ONLY`: phone's real IP (expected — explains to user why IP unchanged)
  - VPN on, `SINGLE_EXIT_NODE`: VPS IP (confirms tunnel is working)
- No continuous polling — user-initiated only.
- Result shown in dashboard; not logged to server.
